## 4. 核心流程

### 4.1 自然语言建规则

```
用户输入一句话
     │
     ▼
NL2RuleService
  ├── 加载对象列表 + 渠道列表 + 指标字典 (作为 LLM 上下文)
  ├── 组装系统提示词 + JSON Schema
  ├── 调用 LLM (function calling / structured output)
  ├── 校验 JSON、字段对齐
  └── 返回填好的 AlertRuleRequest 草稿
     │
     ▼
前端表单回填 → 用户确认 → 保存
```

### 4.2 告警触发与归并

```
MetricSimulator (每 30s)
     │ 生成 mock 指标
     ▼
AlertEngine
  ├── 查启用规则 + 关联对象
  ├── 评估每条规则 (compare op, trigger times, time window)
  ├── 命中 → 进入 IncidentMerger
  └── 未命中 → 跳过
     │
     ▼
IncidentMerger
  ├── 查 Redis: object_id + 30min 内 incident_id
  ├── 命中 → 把新事件挂到已有 incident
  └── 未命中 → 新建 incident
     │
     ▼
EventService.create
     ├── 写 alert_event
     ├── 异步生成 AI 摘要 (写回 event_summary)
     ├── 写 alert_notify_log + 调 NotifySender
     └── SSE 推送给前端
```

### 4.3 AI 告警摘要

```
事件入库后异步触发
     │
     ▼
EventSummaryService
  ├── 装载: 对象画像 + 规则 + 当前值 + 同对象近 24h 历史事件
  ├── Prompt 模板 -> 4 段输出 (what / impact / causes / actions)
  ├── 流式调用 LLM
  └── 保存到 alert_event.event_summary 字段
     │
     ▼
前端详情页订阅 SSE 显示流式生成
```

## 6. 关键设计决策

| 决策 | 选项 | 理由 |
|---|---|---|
| 实时推送 | SSE vs WebSocket | SSE：单向告警推送足够，浏览器友好，无需心跳 |
| LLM 协议 | OpenAI 兼容 | 可换 GPT-4o / Qwen / DeepSeek，灵活 |
| 调度框架 | Spring Schedule | 轻量，告警引擎本地跑就够 |
| ID 生成 | 数据库自增 + 业务编号 | event_no 用 `ALERT-yyyyMMdd-{6位}` 便于检索 |
| 状态机 | 静态枚举 + Service 编排 | 不引入 spring-statemachine，避免过度设计 |
| 表名 | 业务名小写下划线 | 不用 `01_xxx` 前缀，按规范命名 |

## 7. 可扩展点

- **多租户**：所有表预留 `tenant_id` 字段（v2 再激活）
- **指标采集**：当前用模拟器，后续可接 Prometheus / Telegraf
- **更多渠道**：钉钉、飞书、PagerDuty
- **根因分析**：基于 Incident 关联图谱
