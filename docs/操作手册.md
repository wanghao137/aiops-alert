## 1. 系统启动

### 1.1 第一次启动（首次部署）

**前置依赖**：
- JDK 17（必须，Eclipse Adoptium，路径 `C:\Program Files\Eclipse Adoptium\jdk-17.0.19.10-hotspot`）
- Node.js 18+（npm 自带）

**启动**：双击 `aiops-alert/start.cmd`

启动器会：
1. 自检 JDK 17 + npm
2. 启动后端独立黑窗口跑 `mvnw spring-boot:run`
3. 轮询 8090 端口直到后端就绪（最多 90 秒）
4. 首次自动跑 `npm install`（5~10 分钟）
5. 启动前端独立黑窗口跑 `npm run dev`
6. 自动用默认浏览器打开 http://localhost:5173

### 1.2 日常启动（依赖已装）

跟首次一样：双击 `start.cmd`，跳过 npm install，10~15 秒就绪。

### 1.3 启动失败排查

| 现象 | 原因 | 解决 |
|---|---|---|
| 后端窗口报 `JDK 17 未找到` | JAVA_HOME 路径不对 | 改 `start.cmd` 里的 JAVA_HOME |
| 后端窗口报 `file is locked` | 上次后端没退干净 | 先跑 `stop.cmd`，再 `start.cmd` |
| 后端起来但 90s 没就绪 | 启动慢（首次冷编译） | 直接打开浏览器看，后端日志在它自己的黑窗口里 |
| 前端打不开 5173 | 端口被占 | `stop.cmd` → `start.cmd` |
| 浏览器没自动打开 | 浏览器被关闭/默认浏览器没设 | 手动访问 http://localhost:5173 |

### 1.4 停止

双击 `aiops-alert/stop.cmd` —— 按端口（8090 / 5173 / 4173）找进程并 `taskkill /F`。

或者直接关掉对应黑窗口。

## 3. 总览大屏（Dashboard）

### 3.1 大数字 hero

最大那个数字是**当前待处理告警数**。下面一行是组成：紧急 N · 严重 N · 已恢复 N。

**配色**：数字红色 = 有未处理；数字白色 = 全部已处理。

### 3.2 每日 AI 简报

hero 下方紫色卡片。每天 8:00 自动生成，可点 **刷新简报** 手动触发。简报内容：
- 整体态势叙述（200~500 字）
- 昨日 vs 前日变化（百分比箭头）
- TOP 3 重要事件 highlight

### 3.3 KPI 网格

4 个核心指标卡：events / pending / incidents / notify-fail

### 3.4 7 日趋势曲线

四条线：总量 / 待处理 / 已恢复 / 紧急。鼠标悬停看每日数。

### 3.5 NOC 大屏模式

右上角 **大屏模式** 按钮 → 进入：
- 隐藏侧栏 + header
- 显示 "NOC · LIVE" 角标
- 30 秒自动刷新数据

按 **ESC** 退出。

## 4. 告警事件（Events）

### 4.1 三栏布局

| 栏 | 内容 |
|---|---|
| 左侧栏 | 5 个状态 tab：全部 / 待处理 / 已确认 / 已恢复 / 已关闭 |
| 中间 | 事件卡片列表，含级别色条 + AI 摘要小标 + 时间戳 |
| 右侧 | 选中事件时才出现，显示详情 + AI 摘要 + 操作按钮 |

> **没选事件时右侧不占位**，列表撑满中间。

### 4.2 触发演示数据

右上角 **触发演示** 下拉菜单：

- **启动故事模式**：随机选一条规则的对象，让 metric 自然爬升直到命中规则触发告警。30 秒内能看到自动生成事件 + AI 摘要的全过程。**最适合演示**。
- **立即生成测试事件**：手动选规则 + 对象 + 当前值，立即创建一条事件，但跳过自然爬升的剧情感。

### 4.3 AI 摘要状态

| 状态 | 显示 |
|---|---|
| `PENDING` | 紫色"思考中"标签 + 终端打字机动画轮播 5 句调试话术 |
| `SUCCESS` | 紫色"AI"标签 + 摘要正文（包含 root_cause / impact / recommendations 三段） |
| `FAILED` | 红色 ✕ 标签 + 错误信息 + 重试按钮 |

### 4.4 状态机操作

详情右下方 3 个按钮：

```
PENDING ──[确认]──> CONFIRMED ──[恢复]──> RECOVERED ──[关闭]──> CLOSED
```

每次操作都会写 `alert_handle_log` 一条记录，可追溯。

## 6. 告警规则（Rules）

### 6.1 手动新建

右上角 **新增规则** → 弹窗：
1. 基础信息（名称 / 对象类型 / 级别 / 优先级 / 描述）
2. 触发条件（多个 AND/OR + 指标 + 比较符 + 阈值 + AI 阈值推荐按钮）
3. 触发策略（连续 N 次 / 观察窗口 / 最小间隔）
4. 关联监控对象 + 通知渠道

底部 **保存规则** 按钮。

### 6.2 AI 自然语言生成（推荐）

中间面板 **AI BUILDER · BETA**：

```
我想监控生产 MySQL 主库的主从延迟，超过 60 秒就发企微告警
```

回车 → AI 在 30~60s 内生成完整规则结构 → 弹窗预览 → 一键应用。

### 6.3 AI 阈值推荐

弹窗里阈值输入框右侧的 ⚡ 图标 → 调用 `/ai/threshold` 基于 7 日历史 P50/P95 给出推荐值（**低/中/高**三档）。

## 9. 系统设置（Settings · LLM 配置）

### 9.1 LLM 配置卡片

每张卡片显示：模型名 / 提供商 / baseUrl / API Key 掩码 / max_tokens / 温度 / 状态。

DEFAULT 标识 = 业务调用（事件摘要、每日简报、规则生成）走这条。

### 9.2 新增 LLM

右上角 **新增 LLM** → 弹窗顶部 4 个 PRESET 按钮（智谱/DeepSeek/OpenAI/Qwen）一键填入 baseUrl + 默认模型名。

**字段**：
| 字段 | 说明 |
|---|---|
| 配置名称 | 用于卡片显示 |
| 提供商 | OpenAI / ZHIPU / DEEPSEEK / QWEN |
| 模型名 | 例如 glm-5.1 / deepseek-chat / gpt-4o-mini |
| Base URL | OpenAI 兼容 API 端点 |
| API Key | 必填（除非 baseUrl 是本地 ollama） |
| max_tokens | 推荐 ≥ 8192（思考模型要算上 reasoning） |
| temperature | 0.0 ~ 2.0，推荐 0.2~0.5 |
| 默认 | 设为 true 后会取消其他配置的默认 |

### 9.3 测试连通

每张卡片右下 **测试** 按钮 → 调一次极简 prompt（"用 2 个字回答：你好"），10 秒内返回。

错误信息分类：
- HTTP 401/403 → "API Key 无效，或套餐已到期"
- HTTP 404 → "端点不存在：检查 baseUrl 路径与模型名"
- HTTP 429 → "限流：稍后重试或升级套餐"
- Connection reset → "可能是端点不接受此调用方式或瞬时网络抖动"

### 9.4 智谱 GLM Coding Plan 注意

如果用 **GLM Coding Plan 套餐**（编码套餐）：
- baseUrl 必须用 **`https://open.bigmodel.cn/api/coding/paas/v4`**
- 模型名 `glm-5.1` / `glm-4.7` / `glm-4.5-air`
- 套餐有 5 小时限额，达到后会限流

如果用**普通 API 计费**：
- baseUrl 用 `https://open.bigmodel.cn/api/paas/v4`

## 14. 常见问题

### Q1：为什么 AI 摘要要等 60 秒？
GLM-5.1 是推理类模型，思考占大头。EVENT_SUMMARY 通常 40~90s，DAILY_BRIEF 100~130s 是正常的。前端用 PENDING 状态 + 终端打字机动画把等待包装成"演示效果"。

### Q2：刚启动一会 dashboard 数字是 0？
后端冷启动 + 模拟器还没第一次 tick（30s 一次）。等一两分钟数据就齐了。

### Q3：如何重置演示数据？
H2 文件模式存在 `aiops-alert/backend/data/aiops_alert.mv.db`。
- 全停（`stop.cmd`）
- 删 `aiops-alert/backend/data/` 整个目录
- 重启（`start.cmd`）→ Spring Boot 重新跑 schema-h2.sql + DemoDataService.seedHistorical 回填 12 条 7 日历史

### Q4：LLM 调用失败怎么排查？
- 去 `/ai-stats` → 慢调用 / 失败日志
- 看 errorMessage 字段（已经被 LlmClient 包装成友好提示）
- 主要原因：网络抖动 (Connection reset) / 套餐限流 (429) / API Key 失效 (401)
- 客户端已有 1 次自动重试（800ms 退避），瞬态错误大多自愈

### Q5：怎么改 LLM 模型？
`/settings` → 任意配置卡 **编辑** → 改 modelName → 保存 → 设默认。
全局立刻生效，新调用走新配置。

### Q6：每日简报错过 8am 怎么办？
有两种触发方式：
1. 启动时自动 warmup（应用启动后 1 分钟自动跑一次）
2. dashboard 上的 **刷新简报** 按钮手动触发

### Q7：如何加 DeepSeek 模型？
`/settings` → **新增 LLM** → 顶部 PRESETS 点 **DeepSeek Chat** → 填 API Key → 保存 → 测试连通 → 设为默认。

baseUrl 自动填好 `https://api.deepseek.com/v1`，模型名 `deepseek-chat`。

后端代码已经做了 vendor-aware：智谱专有的 `thinking.clear_thinking` 字段只发给 bigmodel.cn，DeepSeek 收不到这个未知字段。

## 16. 故障自救一览

| 现象 | 第一步 | 第二步 |
|---|---|---|
| 浏览器 502 / 拒绝连接 | 看后端窗口是否还活 | `stop.cmd` → `start.cmd` |
| AI 调用一直 PENDING | 查 `/ai-stats` 失败日志 | 检查智谱 API Key + 配额 |
| 弹窗保存按钮看不到 | 已修复（c9487f1）| 拉最新代码 + 刷新浏览器 |
| 事件页右侧大白边 | 已修复（c9487f1）| 拉最新代码 + 刷新浏览器 |
| 数据库被锁 | `stop.cmd` 没把进程杀干净 | `taskkill /F /IM java.exe` |
| 端口被占 | 别的项目抢了 4173/5173/8090 | `Get-NetTCPConnection -LocalPort <端口>` 找 PID 杀掉 |
| 中文乱码 | cmd 默认 GBK | start.cmd / stop.cmd 已用 GBK 编码，子脚本用纯 ASCII |

## 附录 A：技术栈

- **后端**：Spring Boot 3.2 + MyBatis Plus + H2（dev）/ MySQL（prod）+ JDK 17
- **前端**：Vue 3 + Vite 5 + Element Plus + Pinia + ECharts + Tailwind
- **LLM 协议**：OpenAI 兼容（chat/completions），多 vendor 自适应（智谱/DeepSeek/OpenAI/通义/任何兼容代理）
- **实时推送**：SSE（`/stream/alerts`）+ 自动重连
- **构建工具**：Maven Wrapper + npm + Vite

## 附录 B：日志文件位置

- 后端日志：黑窗口实时输出（logback-spring.xml 控制）
- AI 调用流水：`ai_call_log` 表（H2 / MySQL）
- 通知发送日志：`alert_notify_log` 表
- 事件处理日志：`alert_handle_log` 表

## 附录 C：管理员入口

- H2 控制台：http://localhost:8090/api/h2
  - JDBC URL: `jdbc:h2:file:./data/aiops_alert;MODE=MySQL`
  - User: `sa`
  - Password: 空

可以直接 SQL 改数据，**慎用**。

## 附录 D：演示要点（销售/POC 视角）

1. **完整闭环**：监控 → 告警 → AI 摘要 → 通知 → 处理 → 复盘，无人工介入也能完整跑完
2. **AI 原生**：不是事后贴的 Copilot，而是从规则生成、阈值推荐、事件摘要、每日简报、命令面板五个核心环节内生
3. **可观测性**：AI 自己也是一个被监控对象（`/ai-stats` 看 token 成本、慢调用、失败原因）
4. **现代设计**：Editorial × Terminal 设计语言，黑白底色配电光青/sky blue，远超国产监控产品的 UI 水准
5. **演示就绪**：故事模式 / 一键触发 / 预生成历史数据 / NOC 大屏 / 命令面板，每个交互都为演示打磨过
